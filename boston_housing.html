Implementation: Define a Performance MetricÂ¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

# Done: Import 'r2_score'
from sklearn.metrics import r2_score

def performance_metric(y_true, y_predict):
    """ Calculates and returns the performance score between 
        true and predicted values based on the metric chosen. """
    
    # Done: Calculate the performance score between 'y_true' and 'y_predict'
    score = r2_score(y_true, y_predict)
    
    # Return the score
    return score

Question 3 - Training and TestingÂ¶

  • What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?

Hint: Think about how overfitting or underfitting is contingent upon how splits on data is done.

Answer 3:

Splitting the whole dataset into a larger subset (the training set) and smaller subset (the testing set) is a common and recommended practice in machine learning. As names suggest the model is trained on the first set and evaluated on the second one. This practice help to find a model that generalizes well by allowing independent validation on a set of unseen data (Golden rule: Never use the testing data for training).

The ratio training / testing should be large enough in order to prevent underfitting (not enough data to train the model resulting in a poor model) but not too large as this would be in favor of overfitting (tendency of the model to memorize the given data instead of learning the characteristics of it). Common ratios are 75/25 or 80/20 %.

Question 8 - Cross-ValidationÂ¶

  • What is the k-fold cross-validation training technique?

  • What benefit does this technique provide for grid search when optimizing a model?

Hint: When explaining the k-fold cross validation technique, be sure to touch upon what 'k' is, how the dataset is split into different parts for training and testing and the number of times it is run based on the 'k' value.

When thinking about how k-fold cross validation helps grid search, think about the main drawbacks of grid search which are hinged upon using a particular subset of data for training or testing and how k-fold cv could help alleviate that. You can refer to the docs for your answer.

Answer 8:

When tuning the hyper-parameters of a model (with a grid search technique for instance) we need to evaluate the model but we cannot use the testing set for that purpose (remember the golden rule: Never use the testing data for training).

To overcome this, the training set is split itself into another training set and a validation set: the model is trained on the new training set and evaluated on the validation set, and final overall evaluation is done on the test set. But doing so reduce the number of points available for training and the training / validation results can still be affected by a given particular split between the two corresponding sets.

A solution to that issue is cross-validation where the training / validation of the model is performed several times on different splits training / validation of the original training set and the different evaluations are finally averaged. This is what is called cross-validation and the so-called k-fold cross-validation is the most basic technique to achieve that:

  • the original training set is divided into k smaller sets or folds
  • the training of the model is made on (k - 1) folds and evaluated on the remaining unused fold
  • the training / validation process is repeated k times with a different unused fold each time
  • the final training and validation scores are the averaged values of k different processes
  • the final evaluation is made on the testing set (unused so far).

In the case of optimizing a model using grid search, the k-fold cross validation technique can be used to optimize the hyper-parameters based on given range of these parameters, leaving the testing set for the final evaluation.

# Done: Import 'make_scorer', 'DecisionTreeRegressor', and 'GridSearchCV'
from sklearn.metrics import make_scorer
from sklearn.tree import DecisionTreeRegressor
from sklearn.model_selection import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # Done: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # Done: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth': range(1,11)}

    # Done: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # Done: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(regressor, param_grid=params, scoring=scoring_fnc, cv=cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Answer 11:

This model shouldn’t be used in a real-world setting as it has several shortcomings:

  • The model is trained using only 3 features from the original dataset which had 14. Although it is said that the other features have been excluded as non-relevant I think some other features (present or not in original dataset) could help derive a better price model by considering what could influence the price of two houses having the exact same 3 considered features.

  • As such the house itself is described in this model just by 1 feature, its number of room and nothing else. It is obvious that for instance year of construction or overall condition of the house, quality of appliances, surface area of the terrain or presence of a pool have an impact on the selling price. By using only feature for the house I believe we put too much emphasis on the neighborhood irrespective of the quality of the house itself.

  • The neighborhood has clearly an impact on the house’s price but here it is also characterized by only two features which are insufficient to me. Other features like transportation facilities, commercial centers, etc would help to better characterize the environment...

  • The sensitivity test made above clearly shows that the model is not robust enough as it leads to a price range of \$70k for an average price of $400k. Likewise the score of the model is in the order of 0.8 which is good but still leaves place for further improvements.

  • The original dataset was collected in Boston in 1978 and hence any model derived from this would only be applicable to this specific urban area in 1978.

  • Although it is said that the selling price has been scaled to account for 35 years of market inflation, the other parameters LSTAT and PTRATIO have not! That is a severe assumption to believe that nothing has changed locally and take into account only national inflation. Just consider a city like Detroit and its declined over the last 30 years or on the opposite side the evolution of the SF bay area. Furthermore a model applied today on this scaled dataset would still ignore any new construction these past 35 years.

  • No other model than a decision tree has been considered in this project. It would be interesting to see if other models can do better with the same dataset.
